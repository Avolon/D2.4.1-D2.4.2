# D2.4.1-D2.4.2


Kind (Kubernetes IN Docker) — это инструмент для запуска локальных кластеров Kubernetes с использованием Docker контейнеров. Когда сообщество Kubernetes объявило, что Docker будет устаревшим в качестве среды выполнения контейнеров, это вызвало некоторые вопросы и беспокойства среди пользователей. Однако Kind утверждает, что это изменение его не затронет. Давайте разберемся, почему это так.

### Понимание Устаревания Docker в Kubernetes

Чтобы понять, почему Kind не затронет это изменение, важно сначала понять, что означает устаревание Docker в Kubernetes. В контексте Kubernetes, это означает следующее:

1. **Container Runtime Interface (CRI):** Kubernetes взаимодействует со средой выполнения контейнеров через интерфейс CRI. Docker сам по себе не является CRI-совместимым, поэтому Kubernetes использует дополнительный слой под названием **dockershim** для взаимодействия с Docker.

2. **Замена dockershim:** Сообщество Kubernetes решило отказаться от использования dockershim и поддерживать непосредственно CRI-совместимые среды выполнения контейнеров, такие как containerd и CRI-O.

### Почему Kind Не Затронет Это Изменение

Kind не затронет это изменение по следующим причинам:

1. **Использование containerd:** Внутри контейнеров Docker, которые запускает Kind, используется containerd. containerd является CRI-совместимым и будет поддерживаться Kubernetes без необходимости использования dockershim. Таким образом, когда Kind запускает Kubernetes, он фактически использует поддерживаемую CRI-совместимую среду выполнения контейнеров.

2. **Абстрагирование от Docker:** Хотя Kind использует Docker для создания контейнеров, внутри этих контейнеров Kubernetes использует containerd или другую CRI-совместимую среду выполнения. Это означает, что Docker используется только для создания контейнеров, но не как среда выполнения контейнеров для Kubernetes.

3. **Локальная среда:** Kind предназначен для локальной разработки и тестирования. Изменения в производственных кластерах Kubernetes, связанные с удалением dockershim, не влияют на функциональность Kind, поскольку его основной целью является предоставление легкой и быстрой среды для разработки.

### Подведение Итогов

Когда Kubernetes говорит об устаревании Docker, он говорит о Docker как о среде выполнения контейнеров в производственных кластерах. В случае Kind, Docker используется только для запуска контейнеров, внутри которых работает Kubernetes с CRI-совместимой средой выполнения контейнеров, такой как containerd. Поэтому Kind продолжит функционировать без изменений, несмотря на устаревание Docker в производственных кластерах Kubernetes.

Kind в данном случае остается функциональным инструментом для локальной разработки и тестирования, не сталкиваясь с проблемами, вызванными удалением поддержки dockershim в Kubernetes.
